# Phase 2: 작업 단위 로깅 · 플랜 스코어링 · 하네스 트렌드

## Context

Phase 1(훅 경로 수정·디렉토리 구조·JSONL 활동 로그)은 완료됐다. 그러나 **툴 단위 로그는 토이 프로젝트 볼륨으로는 신호가 되기 어렵고**, 사용자는 두 가지 상위 목표를 원한다:

1. **하네스 변경의 효과 측정** — `.claude/`에 수정이 가해진 전후로 작업 품질이 개선됐는지를 정량으로 보기
2. **안티패턴 추출** — 점수가 낮은 플랜들의 공통 실패 요인을 `knowledge/`로 정제

두 목표는 "플랜 1개 = 작업 1개 = 점수 1개" 라는 **동일 데이터 소스**로 해결된다. 플랜 아카이브 파일에 점수·하네스 SHA·작업 유형을 프론트매터로 기록하면 시계열 비교와 저점 클러스터링이 같은 파일 집합 위에서 동작한다.

**양 과적 대응**: 파일 자체는 연 ~2MB로 사소하나 **에이전트 검색 비용**과 **신호 희석**이 실질 문제다. 대응으로 `INDEX.jsonl`(플랜당 1줄 메타데이터) 자동 유지와 최근 30건·90일 슬라이딩 윈도우를 기본 분석 단위로 삼는다.

## Phase 2 스코프 (6개 작업)

1. **데이터 모델 정의** — 플랜 프론트매터 + INDEX.jsonl 스키마
2. **`/task-done` 슬래시 커맨드** — 사용자가 점수 입력하면 아카이빙·인덱싱·상태초기화 일괄 수행
3. **활성 플랜 포인터 자동화** — 플랜 파일 생성·수정 시 `state/current-plan.md`에 포인터 갱신 훅
4. **`/harness-trend`** — 최근 30건·90일 슬라이딩 윈도우로 하네스 SHA별 평균 점수 비교
5. **`/antipattern-scan`** — 점수 ≤2 플랜 3건 이상의 공통 요인을 추출해 `knowledge/antipatterns/` 초안 생성
6. **SessionStart 프라이밍** — 현재 플랜의 작업 유형과 동일한 과거 고점 플랜 상위 3건을 컨텍스트에 주입

## 1. 데이터 모델

**플랜 프론트매터** (Plan 모드에서 작성되는 모든 `.claude/plans/*.md` 첫머리에 YAML):
```yaml
---
task_type: feat     # feat|fix|refactor|docs|test|chore (고정 택소노미)
title: "Phase 2 하네스 스코어링"
created: 2026-04-22
---
```

**아카이브 시 추가되는 필드** (`/task-done` 이 자동 주입):
```yaml
archived: 2026-04-23
score: 4            # 1~5 정수
note: "평가 지표 누락, 다음엔 검증 섹션 보강"
harness_sha: a3f7c1  # .claude/ 마지막 변경 커밋 (7자리)
absorbed: false     # 안티패턴 승격 후 true → 검색/트렌드에서 제외
```

**`INDEX.jsonl`** (`.claude/plans/archived/INDEX.jsonl`, 플랜당 1줄):
```json
{"id":"20260423-harness-scoring","archived":"2026-04-23","task_type":"feat","score":4,"harness_sha":"a3f7c1","title":"Phase 2 하네스 스코어링","absorbed":false}
```

**설계 원칙**: 에이전트는 기본적으로 INDEX 만 읽어 매칭하고, 본문은 최종 후보만 선택적으로 읽는다.

## 2. `/task-done <점수> [메모]` 슬래시 커맨드

**신규 파일**: `.claude/commands/task-done.md`

**동작** (bash 로 실행되는 커맨드 본문):
1. 인자 검증: 점수 1~5, 메모 선택
2. `state/current-plan.md`에서 활성 플랜 파일 경로 읽기 (없으면 에러)
3. `git log -1 --format=%h -- .claude/` 로 하네스 SHA 획득
4. 활성 플랜 파일 프론트매터에 `archived/score/note/harness_sha/absorbed:false` append
5. 파일을 `plans/archived/YYYYMMDD-<slug>.md`로 이동 (slug = title의 kebab-case)
6. `archived/INDEX.jsonl`에 JSON 한 줄 append
7. `state/current-plan.md` 비움 (다음 작업 대기 상태)

**커맨드 본문 구현**: `.claude/scripts/task-done.sh` (python3로 파싱·이동 처리). `.md` 커맨드 파일은 이 스크립트 호출만 담당.

**에러 처리**:
- 활성 플랜 없음 → 에러 메시지 + exit 1
- 점수 범위 벗어남 → 에러 메시지
- 프론트매터 없는 플랜 → `task_type: chore`로 fallback + 경고

## 3. 활성 플랜 포인터 자동화

**신규 훅**: `.claude/hooks/track-active-plan.sh`

**동작** (PostToolUse, matcher `Write|Edit`):
- 수정 파일이 `.claude/plans/*.md` (archived/ 제외)이면 경로를 `state/current-plan.md`에 기록

**효과**: Plan 모드에서 플랜을 쓰거나 수정할 때마다 자동으로 "이게 지금 작업 중인 플랜"이라고 표시됨. `/task-done`은 이 포인터만 신뢰하면 된다.

**드라이버 재사용**: drive-jy `pre-compact.sh` 의 `state/current-plan.md` 관례와 100% 호환.

## 4. `/harness-trend` 슬래시 커맨드

**신규 파일**: `.claude/commands/harness-trend.md` + `.claude/scripts/harness-trend.sh`

**출력 예시**:
```
=== Harness Trend (최근 30건 · 90일) ===

harness_sha  | 커밋 메시지            | 작업수 | 평균점수 | task_type 분포
-------------|----------------------|-------|---------|----------------
a3f7c1 (현재) | feat: scoring system | 8     | 4.1     | feat:5 fix:2 refactor:1
b8e420       | fix: hook stdin bug  | 12    | 3.5     | feat:4 fix:6 docs:2
c2d109       | (Phase 1)            | 10    | 3.2     | feat:3 chore:7

변경 감지: a3f7c1 이후 +0.6점 개선 (표본 n=8, 아직 작음 주의)
```

**로직**:
1. `INDEX.jsonl` 읽어 `absorbed:false` + 최근 90일 + 최대 30건 필터
2. `harness_sha` 로 그룹핑, 평균·건수·유형분포 계산
3. `git show -s --format=%s` 로 각 SHA의 커밋 메시지 병기
4. **표본이 5건 미만인 그룹은 경고** (조기 결론 방지)

## 5. `/antipattern-scan` 슬래시 커맨드

**신규 파일**: `.claude/commands/antipattern-scan.md` + `.claude/scripts/antipattern-scan.sh`

**로직**:
1. `INDEX.jsonl`에서 `score ≤ 2` + `absorbed:false` 플랜 필터
2. 각 플랜의 `note` 필드 + 본문 첫 100줄을 읽어 python `collections.Counter`로 키워드 빈도 집계 (stop words 제거)
3. **3건 이상 공통 키워드**가 있으면 `knowledge/antipatterns/YYYYMMDD-<keyword>.md` 초안 생성
4. 초안 템플릿:
   ```
   ---
   pattern: "<추출 키워드>"
   source_plans: [id1, id2, id3]
   detected: 2026-04-23
   ---
   # 안티패턴: <수동 제목 필요>
   
   관찰된 플랜 N개에서 반복되는 실패 요인:
   - <사용자가 초안 보고 수동으로 요약>
   
   ## 원본 플랜 노트
   - [id1]: <note1>
   - [id2]: <note2>
   ```
5. **승인은 수동** — 사용자가 초안 검토 후 확정하면 소스 플랜들의 `absorbed: true` 로 갱신 (`/absorb <antipattern-file>` 커맨드, Phase 3 확장)

**금지**: 자동으로 knowledge/에 최종본 확정 금지 — 3건 공통이라도 오탐 가능.
