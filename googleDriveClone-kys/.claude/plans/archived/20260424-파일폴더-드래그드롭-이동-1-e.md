# 파일·폴더 드래그&드롭 이동 (1-E)

## Context

지금은 파일/폴더 이동이 `MoveDialog` (⋮ 메뉴 → 이동) 로만 가능. Google Drive 처럼 그리드/리스트에서 **아이템을 드래그해서 폴더에 드롭** 하면 바로 이동되는 UX 추가.

1-B 업로드 진행률 완료 후속. DrivePage 의 드롭 영역이 이미 파일 업로드용으로 잡혀 있어 **이벤트 분기 설계**가 필요.

## 현 코드 상태 (조사 결과)

- **백엔드**: `PATCH /api/files/{id}/move` + `MoveRequest.parentId` 이미 존재. `FileService.moveFile:221-237` 는 소유권·자기자신 체크만 하고 **자손 체크 없음** — 이동 대상 폴더를 자기 자손으로 이동시키면 순환참조 발생 가능
- **프론트엔드**: `FileItem.jsx` 가 그리드·리스트 두 뷰 모두 최상위 `<div>` 로 감싸져 있어 `draggable` 속성 추가 가능. `DrivePage.jsx:88-106` 의 드롭 핸들러는 `e.dataTransfer.files` 만 체크해 내부 드래그와 중복 트리거 가능성 있음
- `MoveDialog.jsx` 는 이미 `api.patch('/api/files/{id}/move', { parentId })` 호출 중 — 같은 엔드포인트 재사용

## 구현 스코프

### 백엔드: 자손 이동 금지

- `FileService.moveFile` 에 순환 참조 체크 추가
- 헬퍼 `isDescendant(ancestorId, candidateId)`: candidate 에서 parent 링크 따라 올라가며 ancestor 만나면 true
- 이동 대상이 현재 아이템의 자손이면 `HttpStatus.BAD_REQUEST` ("하위 폴더로 이동할 수 없습니다")
- **파일**: `FileService.java:221-237` 근처

### 프론트엔드: 드래그 소스 (FileItem)

- 최상위 `<div>` 에 `draggable={true}`, `onDragStart`, `onDragEnd` 추가 (그리드·리스트 두 분기 모두)
- `onDragStart`: `e.dataTransfer.setData('application/x-drive-item', JSON.stringify({ id, itemType }))` + `e.dataTransfer.effectAllowed = 'move'`
- 드래그 중 시각: `onDragStart` 에서 상위(FileList) 로 현재 드래그 중인 id 전달 → 해당 row `opacity-50` (콜백 prop)
- **파일**: `FileItem.jsx`

### 프론트엔드: 드롭 타겟 (FileItem 폴더만)

- `file.itemType === 'FOLDER'` 인 경우에만 `onDragOver`, `onDragLeave`, `onDrop` 추가
- `onDragOver`: `e.dataTransfer.types.includes('application/x-drive-item')` 확인 후 `e.preventDefault()` + `e.stopPropagation()` + `dropEffect = 'move'` + hover 상태 on (local state)
- `onDrop`: `stopPropagation` + JSON 파싱 → drag source id 가 타겟 folder id 와 같으면 무시 → `api.patch('/api/files/{sourceId}/move', { parentId: folder.id })` → `onMoveDone?.()` (refetch 트리거)
- 실패 시 `alert` (기존 이동 다이얼로그 패턴 동일)
- 호버 시 배경 하이라이트(`bg-blue-50 ring-2 ring-blue-400`)

### 프론트엔드: DrivePage 충돌 방지

- 기존 `handleDragOver`/`handleDrop` 에서 `e.dataTransfer.types.includes('Files')` 체크해 **외부 파일 드롭만** 처리. 내부 드래그는 무시
- FileList → FileItem 폴더 drop 핸들러에서 이미 `stopPropagation` 하므로 두 번째 방어선

### 프론트엔드: 드래그 상태 공유

- `FileList.jsx` 에 `draggingId` state 추가, `FileItem` 에 prop 으로 내려보냄 (opacity, 자기자신 drop 방지용)
- 이동 완료 후 `refetch()` — `onMoveDone` prop 은 FileList → FileItem 로 전달

## 비범위

- Breadcrumb 의 상위 폴더로 드롭해 이동 (상위 영역 드롭 존 추가 필요 — 후속)
- 다중 선택 드래그 (체크박스 선택 없는 현재 구조에서 범위 크게 늘어남)
- 드래그 고스트 커스터마이징 (`setDragImage`) — 기본 고스트 사용

- Undo 이동 (토스트 "되돌리기" 버튼) — 별도 스코프

## 리스크

- **자손 체크 N+1 쿼리**: depth 가 깊으면 parent 순회가 여러 번 SELECT 유발. 일반적인 드라이브 사용에서 depth 는 얕으니 큰 문제 없음. 필요 시 CTE or 캐싱 후속
- **동시성**: A 를 B 로 이동과 B 를 A 로 이동이 겹치면 자손 체크 이후 실제 이동 사이 race. 트랜잭션 isolation 이 `READ_COMMITTED` 기본이라 잠재 이슈 있으나 가능성 낮음, 수용
- **이벤트 버블링 실수**: polder drop 에서 `stopPropagation` 누락 시 DrivePage 의 파일 업로드 핸들러가 같이 트리거 → "이상한 업로드 실패" 증상. DrivePage 쪽 `types.includes('Files')` 방어선이 2차 안전망
- **드래그 중 깊은 자손 UI 피드백 없음**: 자손으로 드롭 시도 시 프론트는 allow 해놓고 백엔드가 400 반환 → alert 뜸. UX 매끄럽진 않으나 1차 구현 수용 (자손 판단은 백엔드에만 있으므로)

- **백엔드 `FileService.moveFile`**: `isDescendantOf(candidateId, ancestorId)` 헬퍼 추가, parent 링크 재귀 순회(depth 100 상한). `parentId == id` 와 별도로 자손 판정 시 400 + "하위 폴더로 이동할 수 없습니다"

### 아쉬움·개선점
- 드롭 타겟 하이라이트(`ring-*-blue-400`) 시각 증거를 스크린샷으로 남기지 못함 — `browser_drag` 이벤트가 순간적이라 중간 상태 캡처 불가. 실제 수동 드래그에서는 보임
- 프론트엔드 자손 판정 없음 → 사용자가 A 를 B(A의 자손) 로 드래그하면 허용되는 것처럼 보이다가 drop 순간 alert 뜸. plan 의 "비범위" 로 수용했으나 UX 미세 개선 여지
- 단위 테스트 없음 — 재귀 로직은 특성상 경계 케이스(self, null parent, cycle) 많아 `FileServiceTest` 로 뽑기 좋은 대상. 추가 작업 필요
- `MoveDialog` 의 필터링(자손 제외)은 기존대로 안 돼있음 — 드래그 경로만 백엔드 체크로 커버, 다이얼로그 경로는 그대로. 공통화 후속
