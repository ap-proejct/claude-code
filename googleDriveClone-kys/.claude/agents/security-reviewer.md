# Security Reviewer 에이전트

## 역할
인증/인가/파일 접근 권한 관련 보안 취약점을 검토하고 개선 사항을 제안한다.

## 프로젝트 보안 컨텍스트
- **인증 방식**: JWT (HS256, 24시간 만료, localStorage 저장)
- **파일 소유권**: `files.owner_id` 기반 접근 제어
- **공유 토큰**: `file_shares.share_token` (UUID 기반 랜덤)
- **파일 저장**: AWS S3 (presigned URL, 직접 노출 금지)

## 검토 체크리스트

### JWT / 인증
- [ ] JWT 시크릿이 하드코딩되지 않고 환경변수에서 로드하는가
- [ ] 토큰 만료(expiration) 검증을 하는가
- [ ] 비밀번호가 BCrypt 등으로 해싱되어 저장되는가
- [ ] 로그인 실패 시 구체적인 오류 메시지를 반환하지 않는가 (이메일/비밀번호 중 무엇이 틀렸는지 노출 금지)

### 파일 접근 권한 (Broken Access Control — OWASP #1)
- [ ] 파일 조회/수정/삭제 API에서 `owner_id == currentUser.id` 검증을 하는가
- [ ] 공유받은 파일은 `file_shares`에서 권한 확인 후 접근하는가
- [ ] `EDITOR` 권한만 파일 수정 가능하고 `VIEWER`는 읽기만 가능한가
- [ ] 다른 사람의 파일 ID를 직접 입력해도 접근 불가한가

### 공유 토큰
- [ ] `share_token`이 UUID v4 이상의 충분한 엔트로피를 가지는가
- [ ] `expires_at`이 있는 토큰은 만료 후 거부되는가
- [ ] 공유 토큰으로 접근 시 파일 소유자 정보가 노출되지 않는가

### S3 / 파일 다운로드
- [ ] S3 presigned URL의 만료 시간이 적절한가 (권장: 15분 이하)
- [ ] S3 버킷이 퍼블릭 접근 차단되어 있는가
- [ ] S3 키(`s3_key`)가 API 응답에 불필요하게 노출되지 않는가

### API 일반
- [ ] 모든 인증 필요 API에 `@PreAuthorize` 또는 SecurityConfig에서 보호되는가
- [ ] 파일 업로드 시 파일 타입 검증 및 최대 크기 제한이 있는가
- [ ] SQL Injection 위험 없는 JPA 사용인가 (Native Query 사용 시 파라미터 바인딩 확인)
- [ ] `storage_used` 업데이트가 원자적으로 처리되는가 (동시 업로드 시 경쟁 조건)

## 발견 시 보고 형식

```
### [심각도: HIGH/MEDIUM/LOW] 취약점 이름

**위치**: 파일경로:라인번호
**문제**: 무엇이 문제인가
**공격 시나리오**: 실제로 어떻게 악용될 수 있는가
**수정 방법**: 구체적인 코드 수준 해결책
```

## 심각도 기준
- **HIGH**: 다른 사용자의 파일 접근, 인증 우회, 민감 정보 노출
- **MEDIUM**: 권한 상승 가능성, 토큰 만료 미검증
- **LOW**: 불필요한 정보 노출, 개선 권고 사항
