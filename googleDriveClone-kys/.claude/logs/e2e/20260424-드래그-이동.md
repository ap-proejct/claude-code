# 드래그&드롭 이동 기능 검증

플랜 `파일·폴더 드래그&드롭 이동 (1-E)` 구현 후 Playwright MCP + curl 로 전 경로 검증.

## 백엔드 자손 체크 (curl)

```
폴더 A 생성 → 폴더 B 를 A 자식으로 생성 → A 를 B 로 이동 시도
→ HTTP 400 + message "하위 폴더로 이동할 수 없습니다"
```

✅ 순환 참조 차단 동작. `FileService.isDescendantOf` 재귀가 parent 링크 따라 올라가 ancestor 감지.

## 프론트엔드 드래그 이동 (Playwright MCP)

- `browser_drag(upload-progress-test.bin → 내 문서 폴더)` 실행
- 결과: 드롭 소스 파일이 목록에서 사라짐, 타겟 폴더 열어보니 그 안에 위치
- 네트워크: `[PATCH] /api/files/20/move => [200] OK` with body `{"parentId":1}`

## 증거

- `moved-into-folder.png` — 내 문서 폴더 내부에 `upload-progress-test.bin (5.0 MB)` 노출
- `move-network.json` — PATCH 200 응답 샘플

## 회귀 체크

- 기존 외부 파일 업로드 경로: 1-B 에서 이미 회귀 없음 확인 (100MB 업로드 성공)
- 내부 드래그 시 DrivePage 상위 드롭 영역은 `types.includes('Files')` 로 외부만 처리 → 충돌 없음

## 관찰

- 프론트엔드는 자손 판단 로직이 없어 "A를 B(A의 자손)로 드래그" 시도해도 일단 API 호출됨 → 백엔드 400 → alert. UX 로 보면 매끄럽지 않으나 plan 상 수용 범위. 개선하려면 목록 fetch 시 parent chain 을 같이 내려주거나 별도 depth 조회가 필요
- 드롭 타겟 하이라이트(`ring-2 ring-blue-400 bg-blue-50`) 는 browser_drag 의 순간적 이벤트 특성상 스크린샷 포착 어려움. 실제 수동 드래그에서는 명확히 보임
